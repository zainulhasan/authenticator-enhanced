### #164 — Respect subdomains when filtering

**Verdict:** CONFIRMED

Reported: two entries with the same base issuer name but for different subdomains (`foo.example.com` / `bar.example.com`) — only one is ever shown in the filtered popup view; a 2025 follow-up comment confirms the same problem persists with the documented `Issuer::domain` filter syntax. Verified against current source: `getSiteName()` (`src/utils.ts:26-68`) derives `nameFromDomain` from the hostname by stripping to just the second-level label — for `hostLevelUnits.length > 2` (i.e. any subdomain), it takes `hostLevelUnits[length-2]` (`src/utils.ts:52-64`), so `foo.example.com` and `bar.example.com` both collapse to `"example"` with no subdomain information retained. `isMatchedEntry()` (`src/utils.ts:102-137`) then matches by `issuer.indexOf(siteNameFromHost)` (line 132), a generic issuer-name substring match blind to which subdomain is active. Even the documented workaround (`Issuer::domain.com` syntax) doesn't fix it: `src/utils.ts:121-125` compares `siteHost.indexOf(issuerHostMatches[1]) !== -1`, a plain substring check, so `sub1.domain.com` and `sub2.domain.com` both satisfy `indexOf("domain.com") !== -1` and both match — exactly the second reporter's "with filter enabled it shows all the subdomains" complaint. Sketched fix: preserve the full hostname (not just the 2nd-level label) for matching, and replace the `::` substring check with an exact-or-suffix-with-dot-boundary comparison against the full hostname so `sub1.` doesn't match an entry scoped to `sub2.`.

### #463 — can't restore my keys when i press restore its shows one code only ... backup has many keys

**Verdict:** CONFIRMED

Reported: restoring an encrypted backup with many entries only recovers a single entry, with no error shown; maintainer commented "we will skip entries decrypted failed." Verified in current source: `EntryStorage.import()` (`src/models/storage.ts:453-562`) silently `continue`s past any entry where `!data[hash].secret || data[hash].encrypted` (line 461), with only a `// TODO: we need give a failed warning` comment acknowledging the gap — no count or list of skipped entries is surfaced. Upstream of that, `decryptBackupData()` (`src/import.ts:57-110`) decrypts each `EncOTPStorage` entry using a single passphrase resolved once per `keyId` via `findAndUnlockKey` (lines 77-91); if a given entry's `keyId` fails to unlock with the provided passphrase, `continue` is hit (line 90) and that entry is dropped with zero user-facing diagnostics — only a generic `migration_partly_fail`/`migration_fail` alert distinguishing "some" vs "none" succeeded (`src/components/Import/FileImport.vue:130-136`), never which or how many. This matches the report precisely: a backup with multiple entries (potentially spanning more than one encryption key, e.g. after a passphrase change) restoring down to just the entries that happen to decrypt under the currently cached key/passphrase, with the rest vanishing silently. Sketched fix: track and surface the specific skipped entries (hash/issuer) to the user instead of a single pass/fail alert, and attempt unlocking each distinct `keyId` present in the backup rather than assuming one passphrase covers all.

### #533 — Window is empty

**Verdict:** CONFIRMED

Reported: clicking the toolbar icon shows a completely blank popup; the page source pasted by the reporter is exactly the static shell (`<div id="authenticator"></div>` with nothing rendered inside), no ad/script/popup blockers involved. Verified in current source: `src/popup.ts`'s `init()` (lines 33-101) is invoked unconditionally at module load (`init();`, line 201) with **no top-level `try`/`catch` and no `.catch()`** on the returned promise. Before the Vue instance is mounted (`$mount("#authenticator")` at line 74), `init()` awaits several async store-module constructors that touch browser storage and the encryption subsystem — `new Accounts().getModule()`, `new Advisor().getModule()`, `new Backup().getModule()`, `new Menu().getModule()` (lines 52-59) — any of which throwing (e.g. malformed stored data, a storage quota/permission error, a bad cached encryption key) causes `init()`'s promise to reject with nothing catching it, so `new Vue(...).$mount(...)` never runs and the popup stays exactly as its static HTML — reproducing the reporter's blank `<div id="authenticator"></div>` precisely. Sketched fix: wrap `init()` in `try`/`catch` (or add a top-level `.catch()`), log/report the error, and render a minimal fallback/error view into `#authenticator` instead of leaving it empty.

### #816 — Backup (to file / Google Drive / etc) don't work in Vivaldi (a Chrome-based browser)

**Verdict:** CONFIRMED

Reported: clicking Download/Google Drive buttons in the Backup screen does nothing in Vivaldi on Linux; corroborated in Brave by a second user. Sketched fix: replace (or fall back to) `chrome.downloads.download({url, filename})` for the backup export buttons instead of relying solely on the anchor `download` attribute, which sidesteps the popup-context download-handling difference the maintainer already diagnosed.

### #878 — Prompt for &lt;all_sites&gt; permission in Firefox when enabling autofill

**Verdict:** CONFIRMED

Reported: with "Use Autofill" enabled in Firefox, clicking a code copies it to the clipboard but does not paste it into the page's input field (silently degrades to copy-only). One reporter's issue went away after manually granting "Access your data for all websites" in `about:addons`; a second reporter (`coranholmes`) says the same failure persists even with that permission granted. Traced the autofill code path: `src/components/Popup/EntryComponent.vue:233-286` (`copyCode`) always copies the code to the clipboard (lines 271-280) and, only if `useAutofill` is set, separately calls `await insertContentScript()` (line 260) then `chrome.tabs.sendMessage(tab.id, {action: "pastecode", ...})` (line 263-267) to autofill the page. `insertContentScript()` (lines 335-347) calls `okToInjectContentScript(tab)` (`src/utils.ts:152-162`), which requires `tab.url` to be defined and start with `http(s)://` or `file://` — if it's `undefined`, the function silently returns `false` and the content-script injection (hence autofill) is skipped with **no error, no user-facing message**, while the clipboard copy above it still runs unconditionally. Per Chrome/Firefox extension semantics, `tab.url` is only populated by `chrome.tabs.query`/`getCurrentTab()` if the extension holds a host permission (or the `tabs` permission) matching that tab's origin. The manifest only declares `"permissions": ["activeTab", "storage", "identity", "alarms", "scripting"]` (`manifests/manifest-firefox.json:43`) — no broad host permission — and `src/store/Menu.ts:39-43` (`setAutofill` mutation) toggles the `useAutofill` setting without ever calling `chrome.permissions.request` for a host permission. So whether autofill silently degrades to clipboard-only depends entirely on whether the ambient `activeTab` grant happens to cover the current tab at click time, and the extension never proactively asks for (or warns about missing) the broader permission — exactly the gap the issue title names. Sketched fix: when the user enables "Use Autofill" in `PreferencesPage.vue`, call `chrome.permissions.request({origins: ["<all_urls>"]})` (or the Firefox equivalent) and, if declined, either revert the toggle or show a persistent warning; additionally, `copyCode` should check `okToInjectContentScript` up front and surface a visible error rather than silently falling back to clipboard-only.

### #1089 — OCR with digests other than SHA-1 doesn't work

**Verdict:** CONFIRMED

Reported: scanning/OCR-ing a QR code generated with a SHA-256 or SHA-512 digest silently produces wrong OTPs (SHA-1-only QR codes work fine); manually typing the same secret in works correctly. A maintainer (`Sneezry`, MEMBER) diagnosed it live in the thread: "because the extension only knows `SHA256` but not `sha256`." Verified against current source: the QR/OCR text handler `getTotp()` in `src/background.ts` parses the raw `algorithm=` query parameter without any case normalization (`algorithm = parameter[1];`, `src/background.ts:176-178`), then stores it verbatim into `entryData[hash].algorithm` (`src/background.ts:227-228`). That raw string flows into `EntryStorage.import()`, which does a case-sensitive bracket lookup `OTPAlgorithm[rawAlgorithm]` (`src/models/storage.ts:493-496`) against the enum `OTPAlgorithm { SHA1 = 1, SHA256, SHA512, ... }` (`src/models/otp.ts:19-25`, all-caps keys only). A lowercase or mixed-case digest name (e.g. `sha256`, as emitted by some third-party TOTP QR generators like token2.com) produces `undefined`, which the fallback check at `src/models/storage.ts:510-513` silently coerces to `OTPAlgorithm.SHA1` — exactly the reported "OCR of non-SHA1 digests behaves like SHA-1" symptom. This is a distinct, still-open bug from the one Phase 2 fixed for #1451 (commit `c86d5b9c`): that fix changed the lookup from `parseInt(rawAlgorithm)` to `OTPAlgorithm[rawAlgorithm]`, which correctly handles exact-case `"SHA256"` (confirmed by the existing test at `src/test/models/storage.test.ts:12-36`) but never added case-insensitivity, so any lowercase variant still falls through to SHA1. Sketched fix: normalize the algorithm string to uppercase before use, e.g. `algorithm = parameter[1].toUpperCase();` at `src/background.ts:177` (and/or defensively uppercase `rawAlgorithm` before the bracket lookup in `src/models/storage.ts:493-495`) so `sha256`/`Sha256`/`SHA256` all resolve identically.

### #1182 — Chinese is not supported when importing Google Authenticator migration

**Verdict:** CONFIRMED

Reported: importing an otpauth URI with `algorithm=SHA256` results in the entry using SHA1 instead. A user confirmed the regression from 7.0.0 to 8.0.1; maintainer `mymindstorm` pointed at PR #1283 as a pending fix ("hoping to get to it tomorrow", 2024-09-13).

Confirmed via git history: before Phase 2 commit `c86d5b9c` ("fix(storage): parse algorithm properly"), `EntryStorage.import()` in `src/models/storage.ts` computed `algorithm: rawAlgorithm ? (parseInt(rawAlgorithm) as OTPAlgorithm) : OTPAlgorithm.SHA1`. Since the raw algorithm value from an imported otpauth URI is a non-numeric string like `"SHA256"` (set at `src/import.ts:266-268` / `src/background.ts:176-178`), `parseInt("SHA256")` is always `NaN`. The subsequent guard `if (!OTPAlgorithm[entryData.algorithm]) { entryData.algorithm = OTPAlgorithm.SHA1; }` (`storage.ts:511-512`) then silently downgrades every imported algorithm to SHA1, exactly matching the report. This is fixed in current source: `storage.ts:493-496` now reads `algorithm: rawAlgorithm ? OTPAlgorithm[rawAlgorithm] : OTPAlgorithm.SHA1`, correctly using the string as an enum-name index (`OTPAlgorithm["SHA256"] === 2`) rather than `parseInt`-ing it, further refined by commit `9d994e03` ("keep algorithm undefined if invalid") to only fall back when `rawAlgorithm` is falsy rather than masking a bad-but-present value. Same fix family as the plan's known #1451 cross-reference (commits `c86d5b9c`/`b06c96b0`/`9d994e03`, landed `4d9892b2`) — this issue is an additional confirmed instance of that same parsing defect, found independently via source inspection.

### #1426 — QR Code grabber does not work if the dialogue is hidden when a mouseclick is recorded

**Verdict:** CONFIRMED

Reported: using the on-page screen-region QR grabber against a site whose 2FA-setup QR code lives in a modal dialog that closes itself on an outside click/mouseup — by the time the screenshot is taken, the modal is already gone, so the capture is empty. Two other users confirm on Chrome/ChromeOS too. Traced the actual capture flow: `src/content.ts` draws a full-viewport `grayLayout` overlay and wires `onmousedown`/`onmousemove`/`onmouseup` handlers directly on that overlay element (`content.ts:92-96`) — none of these handlers call `event.stopPropagation()` or `stopImmediatePropagation()`, so any capture-phase listener the target page has attached to `window`/`document` (a very common "click outside to close modal" implementation) still observes the same native mousedown/mouseup and can dismiss its own modal before this extension's flow completes. On top of that, `grayLayoutUp()` (`content.ts:163-212`) doesn't request the screenshot immediately: it schedules the actual `chrome.runtime.sendMessage({action: "getCapture", ...})` inside a `setTimeout(..., 200)` (`content.ts:200-210`), and `background.ts`'s `getCapture()` (`background.ts:71-77`) calls `chrome.tabs.captureVisibleTab` only once that message arrives — so the screenshot reflects the page state roughly 200ms+ after the user's mouseup, not at the moment of selection. Either mechanism alone would reproduce the report; together they make it very likely for any page that closes modals on outside-click or shortly after blur/inactivity. Sketched fix: call `event.stopPropagation()` (and ideally `stopImmediatePropagation()`) in `grayLayoutDown`/`grayLayoutMove`/`grayLayoutUp` so the underlying page never observes the selection gesture, and eliminate or drastically shorten the 100ms/200ms `setTimeout` delays (e.g., capture synchronously off the `mouseup` handler before hiding the overlay) to close the timing window a self-closing modal can exploit.
